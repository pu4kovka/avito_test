# 💰 Balance Service — микросервис для работы с балансом пользователей

Тестовое задание на позицию стажёра-бекендера в юнит **«Работа»**, Avito Tech.

Микросервис централизованно отвечает за баланс пользователей: пополнение,
списание, перевод средств между пользователями и получение текущего баланса
с историей транзакций.

> ⚠️ **Примечание по стеку.** В задании приоритетным языком был указан Go/PHP.
> Это решение выполнено на **Python (FastAPI)** — language-agnostic JSON API,
> описанный в задании, реализован полностью, но если для вакансии принципиально
> важен Go, это стоит обозначить отдельно на собеседовании.

## Задача

> Необходимо реализовать микросервис для работы с балансом пользователей
> (зачисление средств, списание средств, перевод средств от пользователя
> к пользователю, а также метод получения баланса пользователя).

Полный текст задания — см. ниже, раздел [«Исходное ТЗ»](#исходное-тз).

## Модель данных

```sql
-- упрощённо, реальная схема создаётся через SQLAlchemy (Base.metadata.create_all)

CREATE TABLE balances (
    user_id   INTEGER NOT NULL,
    currency  VARCHAR NOT NULL DEFAULT 'RUB',
    amount    INTEGER NOT NULL DEFAULT 0,
    PRIMARY KEY (user_id, currency)
);

CREATE TABLE transactions (
    id           SERIAL PRIMARY KEY,
    user_id      INTEGER NOT NULL,
    type_of      VARCHAR NOT NULL,      -- 'deposit' | 'transaction' | 'transfer'
    currency     VARCHAR NOT NULL DEFAULT 'RUB',
    amount       INTEGER NOT NULL DEFAULT 0,
    description  VARCHAR,
    user_id_2    INTEGER,               -- получатель, заполняется только для transfer
    created_at   TIMESTAMP NOT NULL DEFAULT now()
);
```

Запись о балансе появляется только при первом зачислении — пустых нулевых
строк в таблице `balances` по умолчанию нет, как и требует ТЗ.

## API

Базовый URL: `http://localhost:8000`. Документация Swagger — `/docs`.

### `POST /deposit` — зачисление средств

```bash
curl -X POST http://localhost:8000/deposit \
  -H "Content-Type: application/json" \
  -d '{"user_id": 1, "amount": 500, "currency": "RUB"}'
```

```json
{ "id": 1, "result": "Баланс успешно пополнен на 500.0 RUB" }
```

### `POST /transaction` — списание средств

Используется, когда сервис услуг списывает деньги за покупку.

```bash
curl -X POST http://localhost:8000/transaction \
  -H "Content-Type: application/json" \
  -d '{"user_id": 1, "amount": 200, "currency": "RUB"}'
```

```json
{ "id": 1, "result": "Покупка 200.0 RUB успешно совершена" }
```

Если средств недостаточно, списание не происходит — баланс не может уйти в минус
(проверка `balance.amount < amount` перед списанием).

### `POST /transfer` — перевод между пользователями

```bash
curl -X POST http://localhost:8000/transfer \
  -H "Content-Type: application/json" \
  -d '{"user_id": 1, "user_id_2": 2, "amount": 100, "currency": "RUB"}'
```

```json
{ "id": 1, "result": "Перевод 2 пользователю в размере 100.0 RUB успешно совершен" }
```

Если получателя с таким `user_id_2` не существует (на его счету ещё не было ни
одной операции), перевод не выполняется:

```json
{ "id": 1, "result": "Невозможно сделать перевод - пользователя кому переводят не существует" }
```

### `GET /balance` — получение баланса

```bash
curl "http://localhost:8000/balance?user_id=1"
```

```json
{ "id": 1, "result": "Баланс - [(300.0, 'RUB')]" }
```

### `GET /transaction_list` — список транзакций с фильтрами

Поддерживает пагинацию по сумме/дате и фильтр по типу операции.

```bash
curl "http://localhost:8000/transaction_list?user_id=1&type_of=deposit&lower_amount=0&higher_amount=10000&lower_date=2026-01-01T00:00:00&higher_date=2026-12-31T00:00:00"
```

```json
{
  "id": 1,
  "result": [
    { "type": "deposit", "currency": "RUB", "amount": 500.0 }
  ]
}
```

## Вопросы по ТЗ и принятые решения

Так как список полей и часть деталей в задании были оставлены на усмотрение
кандидата, ниже зафиксированы решения, которые были приняты:

- **Хранение баланса по валютам.** Баланс хранится не одной строкой на
  пользователя, а парой `(user_id, currency)`, чтобы было проще расширять
  систему мультивалютными балансами в будущем — это задел под доп. задание 1.
- **Создание записи баланса.** Запись в `balances` создаётся лениво — при первом
  зачислении, как и указано в ТЗ; до этого баланс пользователя считается отсутствующим.
- **Защита от ухода в минус.** Списание (`transaction`/`transfer`) проверяет
  текущий остаток *до* изменения суммы и выбрасывает исключение `ValueError`,
  если средств не хватает — операция не применяется.
- **Перевод как два метода.** Зачисление и списание объединены в общую функцию
  `update_balances`, а `/transfer` использует её дважды (списание у отправителя,
  начисление получателю) в рамках одной транзакции БД — это гарантирует, что
  баланс не «зависнет» в промежуточном состоянии при ошибке.
- **Проверка существования получателя.** Перед переводом сервис проверяет,
  что для `user_id_2` уже существует строка в `balances` — иначе перевод отклоняется
  с понятным сообщением, а не падает с ошибкой БД.

## Что реализовано из доп. заданий

- ✅ **Доп. задание 2** (список транзакций с пагинацией/фильтрами) — реализовано
  через `/transaction_list` с фильтрами по сумме, дате и типу операции.
- ⬜ **Доп. задание 1** (конвертация баланса в другую валюту через параметр
  `?currency=`) — не реализовано в этой версии; модель данных уже подготовлена
  под мультивалютность, не хватает только интеграции с внешним курсом валют.
